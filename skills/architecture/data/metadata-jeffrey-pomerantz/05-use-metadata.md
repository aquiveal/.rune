@Domain
These rules MUST trigger when the AI is tasked with designing, analyzing, or implementing tracking systems, logging architectures, analytics dashboards, user behavior models, social graph databases, or educational resource usage systems. Activation is strictly required for any system dealing with user interactions, system logs, relationship modeling, or behavioral tracking.

@Vocabulary
- **Use Metadata**: Information capturing how an object, resource, or system has been used, or detailing an individual's behavioral patterns and interactions.
- **Data Exhaust**: A specific subcategory of use metadata representing data produced incidentally as a byproduct of a user doing other things (e.g., purchasing habits, location tracking), rather than data created deliberately.
- **Paradata**: A specialized term for use metadata specifically concerning learning or educational resources (e.g., how often a resource is downloaded, tweeted, or included in curricula).
- **Social Graph / Network**: A simplified structural representation of interpersonal and resource relationships.
- **Node**: An entity within a network. Can be a person, place, or thing (e.g., actor, phone number, email address, IP address, geographic location).
- **Edge**: A defined relationship connecting two nodes (e.g., `was_in_movie_with`, `friend`, `employs`, `travelsWith`, `sentForumMessage`, `calls`).
- **Web Server Access Log**: A fundamental mechanism of use metadata capturing the date, time, requesting application/browser, client IP address, and authenticated user identity.
- **Dashboard**: A presentation layer or tool used to display aggregated use metadata, paradata, or analytics to administrators or educators.

@Objectives
- Precisely model and capture user behavior, resource usage, and interaction histories without conflating them with descriptive metadata.
- Structure complex relational tracking data strictly as mathematical graphs utilizing explicit Nodes and Edges.
- Automate the capture of incidental system interactions as Data Exhaust.
- Enable high-level algorithmic inferences by aggregating discrete, seemingly innocuous data points into comprehensive behavioral profiles.
- Apply domain-specific tracking terminology correctly (e.g., using "Paradata" exclusively within educational/STEM environments).

@Guidelines
- **Graph Structuring**: When mapping interpersonal relationships, communications, or associations, the AI MUST model the data as a network graph. Every entity MUST be an explicitly typed Node, and every interaction MUST be an explicitly labeled Edge.
- **Algorithmic Graph Management Threshold**: The AI MUST assume that any social graph exceeding Dunbar's number (150-250 nodes) cannot be managed or understood by human cognition alone and MUST architect algorithmic management and analysis systems for these networks.
- **Data Exhaust Implementation**: When designing e-commerce, telecommunications, or generic web architectures, the AI MUST architect background collection mechanisms that capture Data Exhaust (incidental data like timestamps, locations, purchased items, viewed items) without requiring deliberate user input.
- **Inference Enablement**: The AI MUST structure Use Metadata such that discrete data points (e.g., individual calls to a home improvement store, a hydroponics dealer, and a locksmith) can be programmatically aggregated to surface behavioral inferences.
- **Educational System Constraints**: When working with learning management systems (LMS) or digital libraries (like the NSDL), the AI MUST classify usage analytics (downloads, shares, modifications) strictly as "Paradata" and treat it as a supplement to, never a replacement for, descriptive metadata.
- **Web Logging Standards**: Any web server architecture generated by the AI MUST include access logs that natively capture: Request Date/Time, Client Application (Browser type/version), Client IP Address, and User ID (if authenticated).

@Workflow
1. **Identify the Tracking Target**: Determine whether the system is tracking an individual's behavior, the usage of a specific resource, or relationships between multiple entities.
2. **Categorize the Metadata**: 
   - If it is educational usage tracking, label and architect it as `Paradata`.
   - If it is background/incidental behavioral tracking, label and architect it as `Data Exhaust`.
   - If it is foundational web traffic, label and architect it as `Web Server Logs`.
3. **Define Network Topologies**: If modeling interactions (like phone records or social links), define the exact categories of `Nodes` (e.g., IP Address, Person) and the exact labels of `Edges` (e.g., `travelsWith`, `purchased`).
4. **Specify Data Points**: Define the exact granular metrics to be captured passively (e.g., time, duration, GPS coordinates, requesting application).
5. **Architect the Aggregation Layer**: Design the output format for the metadata, such as raw access logs for system administrators or visual Dashboards for educators/marketers to draw inferences.

@Examples (Do's and Don'ts)

- **Graph Modeling of Interactions**
  - [DO] Structure communications as a graph: `Node(Type: PhoneNumber, Value: 555-0100)` -> `Edge(Type: calls, Duration: 400s)` -> `Node(Type: PhoneNumber, Value: 555-0200)`.
  - [DON'T] Structure communications as a flat, untyped list: `Call: 555-0100, 555-0200, 400s`.

- **Educational Tracking (Paradata)**
  - [DO] Define LMS analytics as: `Paradata: { resource_id: "physics_module_1", downloads: 450, embedded_in_syllabi: 12, tweets: 30 }` and pair it alongside the resource's descriptive metadata.
  - [DON'T] Store usage metrics in the descriptive metadata fields (e.g., putting download counts in the "Subject" or "Description" fields).

- **Data Exhaust Collection**
  - [DO] Architect e-commerce systems to passively log the sequence of items a user views, the duration spent on the page, and the IP address as `Data Exhaust` for predictive analytics (e.g., predicting a life event like pregnancy).
  - [DON'T] Prompt the user to manually enter a log of the pages they visited or treat deliberate user profile creation as Data Exhaust. 

- **Web Server Log Formatting**
  - [DO] Implement basic Use Metadata logging as: `[2023-10-24 10:00:00] IP: 192.168.1.1 | Browser: Chrome v114 | UserID: 8842 | Request: GET /index.html`
  - [DON'T] Create web logs that only track server errors while omitting client IP, timestamp, and application data.
