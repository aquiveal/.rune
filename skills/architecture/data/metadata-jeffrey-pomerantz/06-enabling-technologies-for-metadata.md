@Domain
These rules trigger whenever the AI is tasked with designing, parsing, structuring, generating, or analyzing metadata schemas, data models, markup languages, semantic web foundations, or any structured dataset utilizing W3C standard architectures (specifically RDF, XML, DTDs, or DOM).

@Vocabulary
- **Structured Data**: Data organized according to a formal data model. All data possesses structure (even natural language text via formatting or statistical distribution), but explicit structure is required for metadata systems.
- **Relational Database**: A system that establishes relationships between fields and tabulated sets of values (controlled vocabularies) to guarantee data quality and prevent ambiguity.
- **Data Model**: A logical representation of the types of entities described by data, their properties, and the relationships between them.
- **RDF (Resource Description Framework)**: A generic W3C data model and framework for making descriptive statements about entities/resources on the web.
- **Triple**: The fundamental 3-part relationship in RDF consisting of a Subject, Predicate, and Object.
- **Graph**: A network formed by a set of interconnected RDF triples where resources act as subjects of some triples and objects of others.
- **URI (Uniform Resource Identifier)**: A unique identifier required for the Subject of an RDF triple so it can be unambiguously referred to.
- **DCMI Abstract Model (Dublin Core Metadata Initiative Abstract Model)**: A generic, modular data model built on RDF that pins down the ontology of metadata schemas, independent of specific syntax.
- **Property-Value Pair**: The combination of a predicate (property) and an object (value) used to describe a resource.
- **Literal Value**: A string of characters that represents an entity (e.g., the text string "Leonardo da Vinci").
- **Nonliteral Value**: An actual entity or resource itself, referenceable via a URI (e.g., the real person Leonardo da Vinci).
- **HTML (HyperText Markup Language)**: A controlled vocabulary for embedding formatting instructions and links into web documents.
- **XML (Extensible Markup Language)**: A set of instructions (structured language) used strictly for specifying and creating *other* markup languages and schemas.
- **DTD (Document Type Definition)**: A document/syntax used to explicitly declare and define all the elements and attributes that exist within a specific markup language.
- **DOM (Document Object Model)**: A hierarchical tree structure containing all elements of a markup language, used by modern web browsers (e.g., for HTML5) in place of DTDs.
- **Technology Stack**: A set of technologies used together to enable specific functionalities; RDF and XML form the base of the W3C stack for the semantic web.

@Objectives
- Transform all implicit structures (like text formatting) into explicit, machine-readable structured data based on rigorous data models.
- Guarantee data quality and disambiguation by enforcing the use of relational structures and controlled vocabularies.
- Model all resource descriptions strictly as RDF Subject-Predicate-Object triples forming interconnected graphs.
- Ensure universal extensibility and interoperability by adhering to the DCMI Abstract Model logic and building custom schemas using XML, DTDs, or DOMs as appropriate to the technology tier.

@Guidelines
- **Structure Recognition**: The AI MUST treat all data as structured. When processing seemingly unstructured text, the AI MUST extract structural clues (e.g., formatting, headers, statistical layout) to map the text to a formal data model.
- **Data Quality Enforcement**: The AI MUST NEVER rely on free-text entry for entities if a controlled vocabulary or name authority file can be applied. Values assigned to shared fields MUST be validated relationally to prevent misspellings and ambiguities.
- **Triple Construction**: Every descriptive statement generated by the AI MUST be reducible to an RDF Triple (Subject, Predicate, Object).
- **URI Mandate**: The AI MUST assign or utilize a URI for the Subject of every triple. Subjects cannot be anonymous strings; they must be uniquely identifiable resources.
- **Literal vs. Nonliteral Strictness**: When assigning values (Objects) in a property-value pair, the AI MUST explicitly distinguish between Literal Values (character strings) and Nonliteral Values (URIs pointing to actual resources). Nonliteral values MUST be preferred whenever the object is an entity that can serve as the subject of another triple.
- **Modular Ontology Design**: When designing a metadata schema, the AI MUST build it modularly using the DCMI Abstract Model logic, ensuring that the schema recognizes standard entities and relationships and can be extended without breaking the base ontology.
- **XML as Schema Foundation**: When tasked with creating a new markup language or a custom metadata serialization format, the AI MUST use XML.
- **Element Declaration (DTD vs. DOM)**:
  - If defining a custom XML schema or working with SGML-based/legacy HTML architectures, the AI MUST declare elements and attributes explicitly using a DTD (Document Type Definition).
  - If generating or targeting modern HTML5 structures, the AI MUST NOT use DTDs for element declaration; instead, it MUST structure elements according to the HTML5 DOM hierarchical tree structure.

@Workflow
When tasked with modeling a resource or defining a new metadata schema, the AI MUST execute the following algorithm:
1. **Entity Identification**: Identify the core resource being described. Assign it a unique URI (establishing it as the Subject).
2. **Property Mapping**: Determine the properties (Predicates) applicable to the resource based on an abstract model (e.g., DCMI).
3. **Value Assignment & Validation**: For each property, determine the Object. 
   - Check if the Object is an entity (Nonliteral) or a string (Literal).
   - If Nonliteral, assign a URI and ensure it links to an authority file or controlled vocabulary.
4. **Graph Construction**: Connect the generated Subject-Predicate-Object triples to form a structured graph. Ensure resources act as both subjects and objects where applicable to build the network.
5. **Syntax Serialization**: Encode the data model into a machine-readable format.
   - Use XML to structure the schema rules.
   - If creating a standalone schema, write the DTD declaring all allowed elements (`<!ELEMENT>`) and attributes (`<!ATTLIST>`).
   - If mapping to HTML5, validate the structure against the modern DOM hierarchy.

@Examples (Do's and Don'ts)

**Principle: RDF Triple Construction and URI Usage**
- [DO] Model a resource description using URIs for subjects and nonliteral objects to form an explicit graph.
  ```xml
  <rdf:Description rdf:about="http://example.com/art/Mona_Lisa">
    <dc:creator rdf:resource="http://authority.example.org/person/Leonardo_da_Vinci"/>
    <dc:title>Mona Lisa</dc:title>
  </rdf:Description>
  ```
- [DON'T] Use ambiguous literal strings for both the subject and the object, preventing network linking.
  ```xml
  <!-- Incorrect: No URIs, treats entities as disconnected literal text -->
  <record>
    <subject>Mona Lisa</subject>
    <predicate>Creator</predicate>
    <object>Leonardo</object>
  </record>
  ```

**Principle: Literal vs. Nonliteral Values**
- [DO] Distinguish clearly between a string representing a name (Literal) and the entity itself (Nonliteral).
  ```text
  Subject: [URI: Mona Lisa]
  Predicate: Creator
  Object (Nonliteral): [URI: Leonardo da Vinci]
  
  Subject: [URI: Leonardo da Vinci]
  Predicate: Name
  Object (Literal): "Leonardo da Vinci"
  ```
- [DON'T] Treat the character string "Leonardo da Vinci" as the actual person who painted the picture within the data model.

**Principle: Schema Definition via DTD**
- [DO] Declare XML metadata elements and their data types rigorously using a DTD when defining a custom markup language.
  ```dtd
  <!ENTITY % customSchema "cs:title | cs:creator | cs:date">
  <!ELEMENT cs:title (#PCDATA)>
  <!ATTLIST cs:title xml:lang CDATA #IMPLIED>
  <!ATTLIST cs:title rdf:resource CDATA #IMPLIED>
  ```
- [DON'T] Output arbitrary XML tags for a new schema without defining their structural rules, relationships, and allowed data types in a DTD (or equivalent modern schema definition).

**Principle: Data Quality via Relational Structure**
- [DO] Restrict field inputs to a tabulated controlled vocabulary to ensure consistency.
  ```json
  {
    "resource": "http://example.com/book/123",
    "format": {
      "value": "application/pdf",
      "vocabulary": "Internet MIME types"
    }
  }
  ```
- [DON'T] Allow unstructured, ad-hoc text entries for fields that represent standardized categories (e.g., allowing "Acrobat File", "PDF document", and "pdf" interchangeably).
