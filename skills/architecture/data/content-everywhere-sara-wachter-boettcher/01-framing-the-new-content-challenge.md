@Domain
Triggered when evaluating, architecting, structuring, modeling, or coding digital content, Content Management Systems (CMS), responsive websites, multi-channel distribution strategies, APIs, or cross-platform data delivery.

@Vocabulary
*   **Content Strategy**: The practice of understanding what content is needed to meet users' needs and organizational goals, producing it, and creating realistic publishing and governance plans.
*   **Fixed Content**: Content that is stuck to individual pages, formatted for a single display (e.g., a desktop monitor), and unable to travel beyond the bounds of a single rendering.
*   **Blob Content (Unstructured Content)**: Words poured onto a page using WYSIWYG (what-you-see-is-what-you-get) editors where formatting (e.g., font size, italics) is used to signify meaning instead of discrete data structure.
*   **Structured Content (Chunks)**: Content broken down and created in smaller, discrete, meaningful modules that can be stored, reconfigured, and used in multiple ways.
*   **Macro Systems**: Traditional website architectures, such as hierarchical site structures and standard navigation submenus.
*   **Micro Systems**: Structures within a single piece of content based on shared attributes and logical rules that allow deep connections, dynamic relation, and reuse across systems.
*   **Future-Friendly**: A design and architectural mindset that acknowledges and embraces the unpredictability of future devices, connected screens, and platforms.
*   **Responsive/Adaptive Design**: A web standards methodology utilizing flexible grids and images to allow layouts to resize and reflow on the fly, abandoning pixel-perfect, fixed-width obsession.

@Objectives
*   Stop creating *new* content for every new platform, channel, or device; instead, architect systems that do *more* with single, centralized content assets.
*   Divorce content meaning, message, and hierarchy from its physical layout and presentation on a screen.
*   Prevent repetitive, manual content management updates (e.g., copy-pasting data between domains) by structuring content for automated reuse.
*   Prepare one set of content to fluidly travel anywhere—from desktops and smartphones to read-later applications, APIs, and undiscovered future devices.
*   Eliminate dead-end pages by connecting siloed data points through shared content attributes and logical display rules.

@Guidelines

*   **Dismantle the WYSIWYG Blob**: The AI MUST NEVER output monolithic blocks of text where HTML styling (e.g., `<b>`, `<i>`, `<font size="+2">`) is the only indicator of a content's role. Visual formatting MUST NOT substitute for structural definition.
*   **Enforce Modular Chunking**: The AI MUST break down narrative content into discrete, identifiable components (e.g., Headline, Teaser, Image, Caption, Subhead, Body).
*   **Eradicate Copy-Paste Workflows**: When requested to duplicate content for a new platform (e.g., "create a mobile version of this text"), the AI MUST refuse manual duplication and instead propose a single structured data model that serves both endpoints.
*   **Abandon the "Page" Paradigm**: The AI MUST NOT architect systems predicated on where content will "live" on a specific URL. The AI MUST architect content based on what it means and how its components relate to each other.
*   **Design for Device Agnosticism**: The AI MUST NEVER assume the final display medium. Content structures MUST NOT be tailored exclusively for iOS, Android, Desktop, or Tablet. The architecture must remain agnostic to support infinite edge cases (e.g., internet-enabled appliances, content-plucking APIs).
*   **Connect the Dots via Shared Attributes**: The AI MUST relate disparate content types (e.g., a "City" and an "Event") using shared metadata attributes and logical rendering rules, rather than relying on manual, hard-coded links or rigid hierarchical menus.
*   **Embrace Layout Unpredictability**: The AI MUST abandon "pixel-perfection" logic. When generating UI/UX code, the AI MUST allow content to reflow, resize, reprioritize, or shift entirely based on the screen context without losing its inherent hierarchy or message.
*   **Maintain Content Integrity**: Stripped of all CSS and formatting, the raw content structure generated by the AI MUST retain its complete story, meaning, hierarchy, and priority.

@Workflow
1.  **Macro/Micro Assessment**: Analyze the provided text or data requirements. Do not ask "What page does this go on?" Instead, ask "What are the salient pieces of information this content is made of?"
2.  **Deconstruction (Chunking)**: Strip away all assumed visual formatting. Break the content down into its atomic semantic elements (e.g., Title, Synopsis, Author, Date, Core Text, Media, Coordinates).
3.  **Attribute Assignment**: Assign metadata and shared attributes to the new chunks. Identify how this content relates to other content in the system (e.g., tying an Article to a Business Listing via a Location attribute).
4.  **Model Generation**: Define a rigid content model (JSON, YAML, or highly semantic HTML/XML) that enforces the separation of these chunks.
5.  **Rule Definition**: Establish logical business rules for how this content should behave (e.g., "If displayed on mobile, extract the Teaser module and prioritize it over the full Body module").
6.  **Future-Friendly Implementation**: Output the structured content or architectural plan, entirely decoupled from CSS/presentational logic.

@Examples (Do's and Don'ts)

**Principle: Structuring Content vs. Blob Formatting**

[DON'T]
```html
<!-- Anti-Pattern: Unstructured WYSIWYG Blob locked to a "Page" paradigm -->
<div class="page-content">
  <span style="font-size: 24px; font-weight: bold;">Sedona Travel Guide</span><br><br>
  <i>By Sarah Krznarich</i><br><br>
  <img src="sedona.jpg" width="600" align="left" style="padding: 10px;">
  Beautiful red rocks and artists' enclaves make Sedona a perfect trip. <br><br>
  <span style="font-size: 18px; color: red;">Upcoming Events:</span><br>
  - Road Biking Tour (Manual Link)<br>
  - Five Star Golf Retreat (Manual Link)
</div>
```

[DO]
```json
// Correct Approach: Agnostic, modular structured chunks ready for multi-channel APIs
{
  "content_type": "Destination_Guide",
  "id": "dest_sedona_01",
  "macro_system_location": "/destinations/southwest/sedona",
  "micro_system_attributes": {
    "location": "Sedona, AZ",
    "theme": ["Outdoors", "Art"]
  },
  "modules": {
    "headline": "Sedona Travel Guide",
    "author": "Sarah Krznarich",
    "teaser": "Beautiful red rocks and artists' enclaves make Sedona a perfect trip.",
    "hero_media": {
      "type": "image",
      "src": "sedona.jpg",
      "alt_text": "Sedona red rock landscape"
    },
    "body_copy": "<p>Famous for its stunning red rocks...</p>"
  },
  "logical_rules": {
    "related_events": "Fetch dynamically where event.location == micro_system_attributes.location",
    "mobile_display": "Prioritize 'teaser' and 'related_events'; collapse 'body_copy' into accordion."
  }
}
```

**Principle: Designing for Fluidity vs. Pixel-Perfect Rigidity**

[DON'T]
```css
/* Anti-Pattern: Hardcoded for a specific device, assuming fixed reading context */
.article-quote {
  position: absolute;
  top: 450px;
  left: 800px;
  width: 300px;
  font-size: 14px;
}
```

[DO]
```html
<!-- Correct Approach: Semantic markup combined with fluid grids, content dictates form -->
<article class="structured-feature">
  <header class="module-headline">...</header>
  <section class="module-body">...</section>
  <!-- The quote is a discrete chunk that can flow anywhere via flexible CSS or be extracted entirely via API -->
  <aside class="module-pullquote" data-relates-to="module-body">
    <blockquote>...</blockquote>
  </aside>
</article>
```
